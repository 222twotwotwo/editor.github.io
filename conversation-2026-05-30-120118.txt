● I now have a full picture. Here's my analysis and plan.

  Why the current approach risks avalanche

  Every write calls INCR posts:ver, which changes the version embedded in every
  cache key at once. So a single like/upload/edit makes the entire posts
  namespace unreachable simultaneously — every list page and every post detail.
  All subsequent reads miss together and stampede MySQL. Orphaned old-version
  keys also linger until TTL.

  Plan (延迟双删 + precise invalidation)

  1. cache/redis.go — remove IncrVersion/GetVersion; add version-free key
  builders, Del, SCAN-based delByPrefix, and InvalidatePosts (delete now → delay
  → delete again, second delete on a background context).
  2. post_handler.go — drop the version from read keys; replace the two
  IncrVersion calls with InvalidatePosts(ctx, docID).
  3. document_handler.go — replace the four IncrVersion calls with
  InvalidatePosts(ctx, id).
  4. Verify → go build ./....

  Key improvement: detail invalidation becomes precise — a like on post A clears
  only post:A, not every post's detail (the main avalanche source). The delayed
  second delete clears any stale value a concurrent read repopulates during the
  write window.

  What changed

  Replaced the global version-number invalidation with 延迟双删 (delayed double

  func (c *Cache) InvalidatePosts(ctx context.Context, docID int64) {
        if c == nil { return }
        c.delPosts(ctx, docID)                    // 1st delete (sync)
        time.AfterFunc(invalidationDelay, func() { // 2nd delete (delayed, 500ms

  Why this addresses the avalanche



  ┌──────────────────┬──────────────────────────┬───────────────────────────┐
  │                  │  Before (version INCR)   │     After (延迟双删)      │
  ├──────────────────┼──────────────────────────┼───────────────────────────┤
  │                  │ Entire namespace — every │ List pages + only the one │
  │ Blast radius of  │  list page and every     │  changed detail           │
  │ one write        │ post detail unreachable  │ (post:{id})               │
  │                  │ at once                  │                           │
  ├──────────────────┼──────────────────────────┼───────────────────────────┤
  │ A like on post   │ Invalidated cached       │ Clears post:3 only; other │
  │ #3               │ details of #1…#N → mass  │  details stay warm        │
  │                  │ miss → DB stampede       │                           │
  ├──────────────────┼──────────────────────────┼───────────────────────────┤
  │                  │                          │ Second delete clears any  │
  │ Stale-on-write   │ —                        │ value a concurrent read   │
  │ race             │                          │ repopulated during the    │
  │                  │                          │ write window              │
  └──────────────────┴──────────────────────────┴───────────────────────────┘

  Two correctness details I want to flag:
  - The second delete runs on context.Background(), not the request context —
  the request context is already cancelled once the response returns, so reusing
  it would make the delayed delete silently fail.
  - The first delete is synchronous (before the response), so the caller's next
  read is immediately fresh; the delayed one is the consistency backstop.

● 延迟双删的实现

  核心是 cache/redis.go 里的 InvalidatePosts，配合各 handler 在更新完 DB 
  后调用它。整体时序是：更新 DB → 立即删缓存 → 延迟 500ms → 再删一次。

  1. 入口方法 InvalidatePosts（redis.go:143）

  func (c *Cache) InvalidatePosts(ctx context.Context, docID int64) {
        if c == nil { return }
        c.delPosts(ctx, docID)                       // ① 第一次删除（同步、立即
        time.AfterFunc(invalidationDelay, func() {   // ② 延迟 500ms 后触发
                bg, cancel := context.WithTimeout(context.Background(), 2*time.S
                defer cancel()
                c.delPosts(bg, docID)                    // ③ 第二次删除（延迟）
        })
  }

  三个关键点：

  - 第一次删除是同步的 —— 在 handler
  返回响应前就执行完，保证写操作的发起者紧接着的读一定是新值。
  - 第二次删除用 time.AfterFunc 调度，不阻塞当前请求。它基于 runtime
  定时器，不是一个 go func(){ sleep } 阻塞的
  goroutine，所以高并发下也不会堆积阻塞协程。
  - 第二次删除必须用 context.Background()，不能用请求的 ctx ——
  这是最容易踩的坑:请求一旦返回,c.Request.Context() 就被 cancel 了,500ms
  后再用它做删除会直接因 context canceled 失败。所以这里新建了一个带 2s
  超时的独立 context。

  2. "删什么"——delPosts（redis.go:130）

  func (c *Cache) delPosts(ctx context.Context, docID int64) {
        c.delByPrefix(ctx, postsListPrefix)   // posts:list:*  —— 列表所有分页
        if docID > 0 {
                c.Del(ctx, PostDetailKey(docID))  // post:{docID}  —— 该贴文详情
        }
  }

  - 列表用 delByPrefix 走 SCAN 游标按 posts:list: 前缀删（offset
  分页下任何增删都会让行跨页移动,所以整组列表页都得清）。用 SCAN 不用
  KEYS,避免阻塞 Redis。
  - 详情是精确删 post:{docID} 这一个 key ——
  这正是相比旧版本号方案的关键改进:点赞 #3 只清 post:3,不波及其它贴文详情。

  3. 它解决的竞态(为什么需要"第二次")

  只删一次扛不住"并发读把旧值回填"的竞态。以给 post #3 点赞为例:

  T1  读请求R：缓存 miss，开始 SELECT（此刻 likes=10）
  T2  写请求W：UPDATE likes=11 并提交
  T3  写请求W：第一次删除缓存          ← 缓存此刻是干净的
  T4  读请求R：把 T1 读到的旧值 likes=10 写回缓存   ← 脏数据出现！
  T5  写请求W：第二次删除(=T3+500ms)   ← 清掉脏值
  之后 读 → miss → 读 DB → likes=11    ✓ 最终一致

  第二次删除就是为了兜底 T4 这个"删完之后又被回填"的窗口。

  4. 延迟值的取舍(redis.go:114）

  invalidationDelay = 500 * time.Millisecond

  它必须略大于一次"读 DB → 
  写缓存"的耗时:只要读请求的回填发生在第二次删除之前,脏值就能被清掉。这是延迟双
  删固有的权衡——延迟太短覆盖不住慢请求,太长则脏值存活窗口变大。另外还有 60s TTL
  作为最终兜底。

❯ 当前能否保证缓存一致性，是否有更优的方案                                      

  Thought for 1m 43s (ctrl+o to expand)

● 先给结论,再讲方案。

  一、当前能否保证一致性?

  不能保证强一致,它保证的是"有上界的最终一致"——上界就是 TTL(60s)。
  延迟双删本质是"尽力而为 + 兜底",不是证明。下面三个场景会出现脏数据:

  场景 1:延迟时间赌输了(尾部风险)

  T1 读R：缓存miss，SELECT 到旧值 V0（此刻写还没提交）
  T2 写W：提交 V1
  T3 写W：第一次删除
  T3+500ms 写W：第二次删除
  T4 读R：因 GC/调度停顿，现在才把 V0 写回缓存（T4 > T3+500ms）  ← 漏网
  只要某个读请求"SELECT → 写缓存"之间停顿超过 500ms,脏值就会越过两次删除,存活到
  TTL 才自愈。500ms 是经验值,不是保证。

  场景 2:第二次删除丢了
  time.AfterFunc 在独立 goroutine 里跑。进程在这 500ms 内崩溃/重启、或那一刻
  Redis 抖动,第二次删除就没了。同样退化到 TTL 兜底。

  场景 3:DB 与缓存非原子
  更新 DB 和删缓存是两个系统,没有跨二者的事务。写请求"提交 DB 成功 →
  还没来得及删缓存就崩了",缓存里的旧值也会留到 TTL。

  ▎ 一句话:常态下 0~500ms 内一致,极端情况下最坏脏 60s(TTL),不会永久脏。

  二、对本项目意味着什么

  帖子列表、点赞数这类数据,最终一致完全够用——它不是账户余额。强一致方案的复杂度
  对一个 Markdown 编辑器的社区流是不成比例的。所以我的整体判断是:当前方案方向正
  确,只需做几处低成本加固,不建议上重型基建。

  三、更优方案(按"本项目性价比"排序)

  ✅ 推荐:低成本、贴合当前规模

  1. singleflight 防击穿(最值得加)
  现在缓存一失效,并发读会同时 miss 并一起砸 DB 重建同一个
  key——这正是你一直关心的"惊群/雪崩"的另一面。用 golang.org/x/sync/singleflight
  把"同一 key 的并发重建"合并成一次 DB
  查询,其余请求等结果。纯进程内、零额外基建、十几行代码。

  2. TTL 随机抖动
  列表被整组清掉后会同时重建、又同时到期(60s 后)形成新的小雪崩。把 60s 改成 60s 
  ± 10% 即可打散。一行的事。

  3. 把易变的 likes_count 从缓存体里剥离(最有针对性)
  关键观察:列表按 updated_at DESC 排序,点赞不改 updated_at,根本不影响列表顺序,只
  改一个显示数字。但现在每次点赞都把整组列表缓存清掉,很浪费。
  更优做法:缓存的 body 里不存 likes_count,读时用项目里已有的 likes:delta
  计数器(IncrLike/GetLikeDelta)+ MySQL 基准值在内存里叠加。这样点赞不再失效任何
  列表/详情缓存——而点赞通常是最高频的写。代价是要把"渲染好的 JSON
  blob"改成"结构体 + 读时拼装",算一次中等重构。

  ⚠️ 更强一致,但对当前规模偏重(我不推荐现在做)

  4. 消息队列做"可靠第二次删除":把第二次删除投递到延迟队列,消费者带重试,解决场景
  2 的丢删。——多引入一套 MQ,收益对本项目不值。

  5. binlog 订阅(Canal/Debezium)驱动失效:订阅 MySQL
  binlog,以"已提交的变更"为准去删缓存,彻底解决场景 3 的双写不一致,且天然可重试。
  这是大型系统里最稳的做法,但要多跑一个订阅服务。结合你 qps.md 里"单实例、14
  行数据、~12K QPS 天花板"的现状,属于明显的过度设计。

  强一致的本质

  真要"读必反映最新写",任何异步失效方案都做不到——只能(a)对这块数据不缓存,(b)走串
  行化的读路径加锁,或(c)缓存+DB
  两阶段提交(不现实)。所以如果某块数据需要强一致,正确答案往往是"别缓存它 /
  用极短 TTL",而不是把失效逻辑越堆越复杂。

  四、我的建议

  保持现在的延迟双删,再加 singleflight 防击穿 + TTL 
  抖动,就是这个项目的甜点位置;如果点赞很频繁,likes_count 
  读时叠加是收益最大的下一步。MQ / binlog 先不碰。

  要的话我可以先把 singleflight + TTL 抖动 加上(改动小、风险低),likes_count
  剥离作为可选的后续重构单独做。你倾向哪个?

  What was built

  Access + refresh tokens layered onto the existing JWT + Redis dual-auth,
  giving silent refresh and self-revocation, per your choices (static RT,
  current-session revocation, keep the per-request whitelist).

  Token model

  ┌─────────────┬─────────────────────────────┬─────────────────────────────┐
  │             │        Access token         │        Refresh token        │
  ├─────────────┼─────────────────────────────┼─────────────────────────────┤
  │ Lifetime    │ 15 min (JWT_ACCESS_EXPIRY)  │ 7 days (JWT_REFRESH_EXPIRY) │
  ├─────────────┼─────────────────────────────┼─────────────────────────────┤
  │ Claim       │ typ:"access"                │ typ:"refresh"               │
  ├─────────────┼─────────────────────────────┼─────────────────────────────┤
  │ Redis       │ jwt:active:{jti} (checked   │ jwt:refresh:{jti} (checked  │
  │ whitelist   │ every request)              │ on refresh)                 │
  ├─────────────┼─────────────────────────────┼─────────────────────────────┤
  │ Purpose     │ API auth                    │ mint new ATs                │
  └─────────────┴─────────────────────────────┴─────────────────────────────┘

  Silent refresh flow — middleware tags an expired AT with error code
  token_expired → frontend interceptor catches it, calls POST /api/auth/refresh
  with the RT (via bare axios, bypassing the interceptor), stores the new AT,
  and replays the original request. Concurrent 401s queue behind a single
  in-flight refresh (isRefreshing + pendingQueue).

  Self-revocation — POST /api/auth/logout (now public) revokes the RT from the
  body and the AT from the header. Since the AT whitelist is checked on every
  request, revocation is instant; the RT being gone blocks further refreshes.
